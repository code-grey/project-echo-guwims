# DevLog: 5 May 2026

## Overview
Today's session focused heavily on expanding the platform to support operational stakeholders: the **Workforce** (Technicians, Cleaners, Security) and the **Admins**. We transitioned the app from a simple reporting tool into a multi-role operational dashboard.

## Architectural Decisions & Real Talk
- **UUIDv4 vs UUIDv7:** We identified that the current PostgreSQL schema uses `gen_random_uuid()` (v4), which can fragment B-Tree indexes at scale. We decided to keep it for this branch to maintain momentum but scheduled a migration to time-sequential UUIDv7 for the next phase.
- **Image Security vs Performance:** We confirmed that our Dart Isolate compression is successfully crushing images to ~600KB, meaning a massive 5MB payload bottleneck doesn't exist. However, the Cloudinary URLs are currently public. We agreed to implement "Authenticated Delivery" (Pre-signed temporary URLs) in the future to protect campus privacy.
- **The "Proof of Work" Strategy:** We discussed the reality of field service tools—without accountability, the system fails. We decided that in the next branch, workers will be *forced* to take a live "After" photo and pass a live GPS distance check (`ST_Distance` <= 50m) before they can mark a task as `RESOLVED`.
- **Dispatch Medium:** We need to settle on the actual dispatch workflow in the real world. Will tasks only appear in the app's Queue UI, or do we need to integrate SMS (Twilio) or Email alerts for critical infrastructure failures? This remains an open question for the next session.

## Backend Operations (Go / PostgreSQL)
- **BOLA Enforcement:** Fixed a critical visibility bug. The `GET /api/reports/nearby` endpoint now strictly filters by `ReporterID` if the user is a `STUDENT`, preventing them from seeing other students' reports. Admins and Workers retain global/departmental visibility.
- **Sorting:** Updated all PostGIS queries to include `ORDER BY created_at DESC` so the newest reports appear at the top.
- **Workforce Queue:** Implemented `GetReportsByDepartmentAndStatus` and the `GET /api/reports/queue` endpoint to serve departmental tasks to workers.
- **Admin Management:** Built the `AdminHandler` with full CRUD operations for users, including a backend parser for bulk CSV imports (`POST /api/admin/users/import`).
- **Worker Resolution:** Updated `UpdateStatus` BOLA checks to allow Workers (not just Admins) to change a report's status to `RESOLVED`.

## Frontend Operations (Flutter)
- **State Management Turmoil:** Attempted to upgrade Riverpod providers to the 3.x `Notifier` API, but encountered massive compilation failures because the project's setup relies on the older `StateNotifier` paradigm. We rolled back to `StateNotifier` to ensure build stability.
- **Workforce UI:** Built `WorkforceQueueScreen`. Technicians now log in and are routed directly to a clean list of their departmental tasks with a "Resolve" button and thumbnail previews.
- **Admin Dashboard:** 
  - Added Material `ChoiceChip` filters (ALL, LOGGED, ACTION_REQUIRED, DISPATCHED, RESOLVED) to the main dashboard.
  - Built the `AdminUserScreen` allowing manual creation and deletion of users based on their role.
- **Local Dev Networking:** Discovered the Android physical device was failing to connect because the host PC's IP changed. Updated `dio_client.dart` to dynamically use the correct IPv4 address.
- **Aesthetics:** Explicitly enabled `useMaterial3: true` in `app_theme.dart` for modernized rounding and components.
