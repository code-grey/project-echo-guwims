# DevLog: 3 May 2026

## Overview
Today's session focused heavily on bridging the gap between the theoretical backend scaffolding created previously and a fully functional, end-to-end mobile application. We established the complete Flutter data layer, integrated the camera and location pipelines, implemented a robust security/routing model via the Gemini API, and finalized the UI polish for both Students and Admins.

## Phase 1: Core Mobile Foundation
*   **Architecture:** Implemented a feature-driven Hexagonal/Clean architecture directory structure (`core/`, `features/auth`, `features/grievance`).
*   **Networking:** Configured `Dio` as the network client. Built a critical Interceptor to handle the dual-token JWT authentication flow. It seamlessly catches `401 Unauthorized` errors, pauses the request queue, hits `/api/auth/refresh`, and replays the requests without dropping the user session.
*   **State Management:** Chose `flutter_riverpod` for compile-safe, completely null-safe dependency injection and state tracking (`AuthNotifier`, `ReportNotifier`).

## Phase 2 & 5: Camera, Location, and Upload Pipeline
*   **Location:** Integrated `geolocator` to handle Android runtime permissions and fetch high-accuracy `latitude/longitude` coordinates. (Fixed a bug by adding `ACCESS_FINE_LOCATION` to the Android Manifest).
*   **Media Compression:** Integrated `image_picker` and `flutter_image_compress`. Crucially, we enforced `keepExif: true` during the 80% compression step so the payload stays under the backend's 5MB limit without stripping geographic metadata.
*   **Form Upload:** Configured the `GrievanceRepository` to send `multipart/form-data` containing the image and a JSON-encoded `metadata` string to `POST /api/reports`.

## Phase 4 & 6: Backend Wiring & Endpoint Completion
*   **Database Fixes:** Addressed an architectural pivot. Instead of a full `username` migration, we updated the Flutter payload to map to the established `university_id` column.
*   **Schema Update:** Made `pin_hash` nullable in Postgres and added `auth_provider` / `provider_id` columns to seamlessly support a future OAuth migration.
*   **Missing Endpoints Built:**
    *   `POST /api/auth/refresh`: Implemented the logic to accept a 6-month Refresh Token and issue a new 15-minute Access Token.
    *   `DELETE /api/reports/delete`: Implemented a hard delete feature. It strictly enforces BOLA (Broken Object Level Authorization) so only Admins or the original reporter can trigger it. It also safely deletes the image blob from Cloudinary to prevent orphaned storage costs.

## Phase 8: AI Routing, Security, and UI Polish
*   **Prompt Injection Defense:** Hardened the Gemini AI prompt to prevent students from abusing the free tier (e.g., trying to solve math equations). It now forces a JSON return of `INVALID_IMAGE` if the picture is unrelated to campus infrastructure.
*   **Dynamic Department Routing:** The Gemini prompt now forces the AI to categorize the issue into `CIVIL`, `ELECTRICAL`, `ESTATE`, or `UNKNOWN`.
*   **UI Upgrades:**
    *   **Caching:** Implemented `cached_network_image` to drastically reduce Cloudinary bandwidth and improve list scrolling performance.
    *   **Timezones:** Added `.toLocal()` to format the Postgres UTC timestamps perfectly for the user's specific timezone (IST).
    *   **Admin Controls:** The `ReportDetailSheet` now displays the reporter's ID (via a newly created SQL `JOIN`). Admins have Dropdown menus to explicitly override the AI's Department routing and freely change the report Status.
    *   **Reporter Edits:** Students can tap a pencil icon to override the AI description manually, hitting a new `PUT /api/reports/update` backend endpoint.

*   **Health & Uptime:** Added a lightweight `GET /healthz` endpoint. This sits outside the JWT middleware so it can be hit directly by uptime monitors.
*   **Self-Pinging Engine:** Updated the background goroutine in `main.go` to hit the `/healthz` endpoint every 10 minutes (using a `LIVE_URL` env variable) to strictly prevent Render's free tier from sleeping.
*   **Architecture Decision (Inference):** Officially rejected on-device ML inference (TFLite). While possible, loading a Vision-Language Model on-device would bloat the app from ~30MB to over 1GB, drastically harming performance. We are strictly relying on the Cloud API (Gemini-2.5-flash).
