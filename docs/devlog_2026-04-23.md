# DevLog: 23 April 2026

## Overview
Today, we laid the foundational architecture for the Go 1.26 monolithic backend for Project Echo (GU-WIMS). We established a strict Hexagonal Architecture (Ports and Adapters) and implemented the core authentication and ticket management workflows, adhering to the security and performance constraints outlined in the project context.

## Architectural Setup
*   **Structure:** Created the standard `cmd/`, `internal/core/` (domain, ports), and `internal/adapters/` (handlers, repository, storage) directories.
*   **Routing:** Utilized the standard library `net/http` multiplexer.
*   **Environment:** Integrated `godotenv` to load configuration from `.env` files (e.g., `DATABASE_URL`, `JWT_SECRET`, `CLOUDINARY_URL`), ensuring no hardcoded secrets remain in the codebase.
*   **Keep-Alive:** Added a background goroutine in `main.go` that pings every 14 minutes to prevent Render from sleeping the instance.

## Database & Data Models (PostgreSQL 17 + PostGIS 3.5)
*   **Schema Design:** Wrote `sql/schema.sql` defining `users` and `tickets` tables. Enforced PostGIS spatial types (`geometry(Point, 4326)`) and `TIMESTAMPTZ` for dates.
*   **Identity Shift:** Made a critical pivot from using `roll_number` to a generic `username` field in the `users` table to support both students and admins without duplicating authentication flows.
*   **Enums:** Defined database enums for `user_role` ('STUDENT', 'ADMIN', 'DISPATCHER') and `ticket_status` ('REPORTED', 'CLUSTERED', 'DISPATCHED', 'RESOLVED').
*   **sqlc Integration:** Configured `sqlc.yaml` to generate type-safe Go code using `pgx/v5`. Wrote PostGIS-aware queries in `sql/query.sql` (using `ST_DWithin` and returning lat/lon via `ST_X`/`ST_Y`).
*   **Domain Models:** Created pure Go structs in `internal/core/domain/` for `User` and `Ticket`.
*   **Repositories:** Implemented `UserRepo` and `TicketRepo` in `internal/adapters/repository/postgres/`, wiring the generated `sqlc` queries to the domain interfaces.

## Business Logic & Handlers
Built three primary HTTP handlers with strict rule enforcement:

1.  **Auth Handler (`/api/auth/login`):**
    *   Accepts `username` and `pin`.
    *   Validates against the DB using `bcrypt.CompareHashAndPassword`.
    *   Generates a 15-minute JWT Access Token.
    *   Sets a 6-month HttpOnly Secure Refresh Token via cookies.

2.  **Ticket Creation (`POST /api/tickets`):**
    *   **Payload Protection:** Enforced a strict 5MB limit using `http.MaxBytesReader`.
    *   **Form Parsing:** Set up `ParseMultipartForm` to handle image uploads alongside form data (title, description, lat, lon).
    *   **Security:** Implemented Magic Byte scanning on the first 512 bytes of the uploaded file via `http.DetectContentType` to ensure only valid images are processed.
    *   **Concurrency:** Designed to return a `202 Accepted` response instantly.
    *   **Background Processing:** Spawns a goroutine to handle the slow operations: uploading the image to Cloudinary (via `cloudinary-go/v2`) and saving the ticket record to PostgreSQL.

3.  **Ticket Operations:**
    *   `GET /api/tickets/nearby`: Fetches tickets within a specific radius using PostGIS `ST_DWithin`.
    *   `PUT /api/tickets/update-status`: Updates the status of an existing ticket.

## Middleware & Routing
*   **Auth Middleware:** Created middleware that extracts the `Bearer` token from the `Authorization` header, verifies the JWT signature, extracts the `sub` (user ID), and injects it into the request context.
*   **Router:** Consolidated all endpoint registrations into `internal/adapters/handlers/http/router.go` for cleaner organization in `main.go`.
